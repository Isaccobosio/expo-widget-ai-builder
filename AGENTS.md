# AGENTS.md — Expo 56 Widget AI Builder

> Multi-agent operating model for the **Fatture in Cloud Widget POC**.
> Stack: **Expo SDK 56 + TypeScript + the official [`expo-widgets`](https://docs.expo.dev/versions/latest/sdk/widgets/) + [`@expo/ui`](https://docs.expo.dev/versions/latest/sdk/ui/) + a local MCP mock server**.
> Target: **iOS and Android simulators in debug mode**. No EAS, no signing, no store artifacts.

---

## 0. Global Constraints (apply to every agent)

- **Expo SDK 56 is mandatory.** Never bump to SDK 57+. All `expo` / `expo-*` packages MUST stay on their SDK 56 versions (use `npx expo install --check --fix` — never `npm i` for Expo modules).
- **Only official Expo modules.** The widget stack is:
  - `expo-widgets` (config plugin + JS API `createWidget` / `updateTimeline`)
  - `@expo/ui/swift-ui` (widget UI components — `VStack`, `HStack`, `Text`, `Spacer`, `modifiers.*`)
  - No third-party widget libraries.
- **No hand-written native code.** All Swift, Kotlin, XML, entitlements, `Info.plist` entries and Xcode target settings are generated by the `expo-widgets` config plugin during `npx expo prebuild`. Agents MUST NOT edit files under `ios/` or `android/` — those folders are regenerated on every prebuild.
- **Zero-Touch execution.** Do not ask the user to run commands, paste code, or edit files. Every action must be performed by an agent tool call.
- **Debug only.** No `eas.json`, no production build flags, no Apple/Google signing steps.
- **No secrets.** The MCP server is a local mock. Never call the real Fatture in Cloud API. Never store credentials.

## 1. Roles

### 1.1 🎯 Orchestrator (PM)

**Purpose:** Owns the roadmap, decomposes user intent into tasks, dispatches work, reports progress.

**Responsibilities**

- Read `README.md`, `.roadmap/`, and the user's latest prompt. Map it to one of the 5 execution phases.
- Maintain a short todo list of the current phase; check items off as they complete.
- Route work: research → **Tech Lead**, code changes → **Coder**, verification → **Reviewer**.
- Never write code, never run build commands directly.
- Escalate to the user only for **hard blockers** (missing simulator, license acceptance, hardware permission).

### 1.2 🧠 Tech Lead

**Purpose:** Owns technical direction. Reads the source of truth for every dependency before the Coder writes anything.

**Responsibilities**

- Consult authoritative docs first: `docs.expo.dev/versions/latest/sdk/widgets`, `docs.expo.dev/versions/latest/sdk/ui`, MCP SDK. Never rely on memory for version-specific APIs.
- Produce **implementation prompts** for the Coder that include: exact file paths, exact package versions, exact config-plugin arguments, and platform-specific gotchas (App Groups on iOS, Android `enableAndroid: true`, iOS deployment target `16.4+`).
- Decide the widget's data-flow contract: what shape the app pushes via `Widget.updateTimeline([...])`, when it re-pushes.
- Own version pinning in `package.json` — always let `npx expo install` resolve, never hardcode.

### 1.3 🛠️ Coder (Dev)

**Purpose:** Executes the Tech Lead's spec. Writes TypeScript only.

**Responsibilities**

- Configure the plugin in `app.json` (`expo.plugins["expo-widgets"]`).
- Author the widget file under `src/widgets/*.tsx` using the `'widget'` directive and components from `@expo/ui/swift-ui`.
- Wire the JS layer: fetch from the MCP mock (`src/data/mockClient.ts`), format via `src/data/formatWidgetProps.ts`, call `widget.updateTimeline([...])`.
- Maintain the MCP mock server under `mcp-server/`.
- Run `npx expo prebuild --clean`, `npx expo run:ios`, `npx expo run:android` and forward logs to the Reviewer.
- **Never** hand-edit `ios/` or `android/` — regenerate via `npx expo prebuild --clean` instead.

### 1.4 🔍 Reviewer (QA)

**Purpose:** Gatekeeper. Nothing ships to the simulator until it passes.

**Responsibilities**

- Before every native build:
  - `npx expo-doctor` — must pass 18/18 checks.
  - `npx tsc --noEmit` — must be clean.
  - `npx expo prebuild --clean --no-install` — must finish without errors.
- After every failed build: parse the terminal log, identify the top actionable error, and hand a **precise fix instruction** back to the Coder.
- If the failure is inside `ios/` or `android/` generated code, the fix is almost always in `app.json` (plugin config) or in the widget TSX — not in the native file itself. Regenerate via prebuild after every fix.
- Confirm success criteria at end of Phase 5: (a) build green, (b) widget visible on simulator home screen, (c) widget renders values coming from the MCP mock.

## 2. Handoff Protocol

Every handoff message between agents must include:

1. **Phase** (1–5 from `README.md`).
2. **Goal** — one sentence.
3. **Files touched or to touch** — absolute paths.
4. **Acceptance check** — what the Reviewer will verify.

Example:

```
Phase: 4
Goal: Add updatedAt footer to the widget.
Files: src/widgets/FatturatoWidget.tsx
Acceptance: Prebuild is clean, widget renders "aggiornato HH:mm" at the bottom.
```

---

## 3. Data Contract (App JS → Widget)

The app owns fetching from the MCP mock and formatting numbers/dates. The widget receives only strings, so its runtime is deterministic and locale-agnostic.

`FatturatoWidgetProps`:

```ts
{
  month: string; // e.g. "Luglio 2026"
  fatturato: string; // e.g. "€ 18.420"
  scadenze: Array<{
    id: string;
    label: string; // truncated to <= 28 chars
    due: string; // e.g. "10 lug"
    amount: string; // e.g. "€ 2.450"
  }>;
  updatedAt: string; // e.g. "aggiornato 12:04"
}
```

Push mechanism: `fatturatoWidget.updateTimeline([{ date: new Date(), props }])`.

Shared iOS App Group: `group.com.teamsystem.fic.widgetpoc` (declared in `app.json > expo.plugins["expo-widgets"].groupIdentifier` — the plugin wires up entitlements automatically).

## 4. Definition of Done (Phase 5)

- [x] `npx expo-doctor` → 18/18 pass.
- [x] `npx tsc --noEmit` → clean.
- [x] `npx expo prebuild --clean` → finishes without errors and generates `ios/ExpoWidgetsTarget/FatturatoWidget.swift` + `android/app/src/main/java/.../FatturatoWidgetProvider.kt` + `android/app/src/main/res/xml/fatturato_widget_info.xml`.
- [ ] `npx expo run:ios` → app launches on iOS Simulator, widget can be added from Home Screen, values come from the MCP mock.
- [ ] `npx expo run:android` → app launches on Android emulator, widget can be added, values come from the MCP mock.

---

## 5. Non-Goals (out of scope for this POC)

- Real Fatture in Cloud API integration or OAuth.
- Live Activities / Dynamic Island (the module supports them, but the POC is home-screen only).
- iOS interactive widgets (buttons/toggles) — read-only only.
- Widget configuration UI (`IntentConfiguration`) — static only.
- Production builds, EAS, TestFlight, Play Console.
- Manual Swift / Kotlin — everything native is plugin-generated.
